### 四、特殊情况处理指南

#### 4.1 双许可证软件
- 优先选择宽松许可证选项
- 确保选择的许可证适用于我们的使用场景
- 在软件清单中明确记录选择的许可证

#### 4.2 许可证不兼容情况
- 不同许可证的软件组合使用时需评估兼容性
- 参考兼容性矩阵进行判断
- 必要时咨询法务专业意见

#### 4.3 许可证变更处理
- 关注软件许可证变更公告
- 评估新许可证对我们的影响
- 制定相应的处理方案

**开放要求：**

### 六、工具配置指南

#### 6.1 CI/CD集成示例
```yaml
# GitHub Actions示例
name: Security Scan
on: [push, pull_request]
jobs:
  security-scan:
    runs-on: ubuntu-latest
    steps:
    - uses: actions/checkout@v2
    - name: Run OWASP Dependency Check
      uses: dependency-check/Dependency-Check_Action@main
    - name: Run Snyk
      uses: snyk/actions/node@master
      env:
        SNYK_TOKEN: ${{ secrets.SNYK_TOKEN }}
```

#### 6.2 定期扫描配置
```bash
#!/bin/bash
# 每日自动扫描脚本
DATE=$(date +%Y%m%d)
REPORT_DIR="/reports/$DATE"

# 创建报告目录
mkdir -p $REPORT_DIR

# 运行漏洞扫描
dependency-check --project "MyProject" --out $REPORT_DIR --format ALL

# 运行许可证扫描
scancode -clpie --json $REPORT_DIR/license_scan.json ./src

# 发送报告
python send_report.py $REPORT_DIR
```

#### 7.1 选型原则
1. **功能需求匹配**：工具功能与实际需求匹配度
2. **成本效益**：考虑购买成本和维护成本
3. **集成难度**：与现有开发流程的集成复杂度
4. **准确度**：误报率和漏报率平衡
5. **更新频率**：漏洞库和规则的更新及时性

#### 7.2 评估清单
在选择工具时，建议评估以下方面：
- [ ] 支持的编程语言和包管理器
- [ ] 漏洞数据库的完整性和时效性
- [ ] 许可证识别的准确性
- [ ] 报告格式和可读性
- [ ] API集成能力
- [ ] 技术支持和文档质量
- [ ] 供应商的可靠性和持续性
